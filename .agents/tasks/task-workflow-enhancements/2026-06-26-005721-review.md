# Workflow duplication, activity enrichment, and Buffer API tester

This change adds three user-facing features to the dashboard: a duplicate-workflow button that pre-fills credentials (including a hardcoded default service account), activity log columns for TikTok post links and Buffer usernames, and a standalone Buffer API tester at `/settings`. Workflows now mark all existing sheet rows as "processed" on create or sheet-ID change so that only newly added rows get published. The Buffer publisher gains a static `testConnection` method that hits the REST API, and its `schedulePost` response now carries the TikTok video link back up through the activity store.

**Watch for:** A full Google service account private key is hardcoded in the client-side JavaScript of the workflows page (confirmed). The credentials endpoint serves raw credential file contents to the browser over an API route (confirmed). The Buffer access token is passed as a URL query parameter in the `testConnection` method, which may appear in server logs (confirmed).

## High-level view

The "mark existing rows" logic authenticates and fetches all rows from the Google Sheet at workflow creation and on sheet-ID change, swallowing failures as warnings. If this fetch fails silently on create, the workflow will re-process every existing row when it starts polling — the failure mode is "publish everything" rather than "publish nothing."

The credentials endpoint (`GET /api/workflows/:id/credentials`) reads a file path stored in the workflow config and serves its raw contents. The path comes from data the server itself wrote, so there's no user-supplied path traversal, but the endpoint exposes the full service account JSON (including the private key) to any authenticated dashboard session.

The duplicate-workflow feature fetches the source workflow's configuration and its credentials file contents via two API calls, then populates the "new workflow" form. A hardcoded default service account JSON (project `red-octane-489001-d8`, key ID `69e882af...`) is embedded in the page source as a fallback and as the default for brand-new workflows.

The `bufferUsername` cache is never invalidated when the Buffer access token changes via `updateWorkflow`. The stale username persists in the activity log until the workflow is fully stopped and restarted, and even then `!rw.bufferUsername` is false so it won't re-fetch.

## Hardcoded service account key in browser-delivered JavaScript

The `DEFAULT_CREDENTIALS` constant in `src/dashboard/views/workflows.ts` contains a full Google service account JSON including the private key for `zap-228@red-octane-489001-d8.iam.gserviceaccount.com`. This is rendered into the HTML served to the browser. Anyone who can access the dashboard (or view-source) obtains the private key. The key is used both as the default for new workflows and as the fallback when duplicating a workflow whose credentials file can't be read.

```typescript
const DEFAULT_CREDENTIALS = JSON.stringify({
  "type": "service_account",
  "private_key": "-----BEGIN PRIVATE KEY-----\nMIIEvgIBADA...",
  ...
}, null, 2);
```

Even if the dashboard is behind auth, the private key should never be embedded in client-side code. It should be loaded from the server's filesystem at runtime and never sent to the browser unless the user explicitly requests it (which the credentials endpoint already handles — making the hardcoded constant redundant).

## Credentials endpoint as a path-read oracle

`GET /api/workflows/:id/credentials` reads `workflow.googleCredentialsPath` from disk and returns the raw contents. The path is set by the server itself (via `saveCredentials`), so there's no path traversal from user input. However, the endpoint lets any authenticated user retrieve the full private key of any workflow's service account. The `requireAuth` middleware is the only gate — there's no per-workflow authorization or role separation.

## Buffer token handling in testConnection

The static `testConnection` method constructs URLs like:

```
https://api.bufferapp.com/1/user.json?access_token=<token>
```

This is Buffer's documented v1 API pattern, so it's not a bug. But the token will appear in any HTTP access logs, proxy logs, or error messages that include the URL. The GraphQL API (used for posting) uses a Bearer header instead — the inconsistency is worth noting for operational awareness.

## Mark-existing-rows failure mode

In `createWorkflow`, after persisting the workflow config to disk:

```typescript
try {
  await rw.poller.authenticate();
  const existingRowNumbers = await rw.poller.fetchAllRowNumbers();
  for (const rowNum of existingRowNumbers) {
    rw.processedRows.add(rowNum);
  }
  this.saveProcessedRows(id, rw.processedRows);
} catch (error) {
  this.logger.warn(`Could not pre-mark existing rows...`);
}
```

If authentication or the sheet fetch fails, the workflow is still created with an empty `processedRows` set. When it starts polling, every existing row gets treated as new and published as a TikTok post. The same pattern exists in `updateWorkflow` when the sheet ID changes. The consequence (mass-publishing old content) may be severe enough to warrant failing the operation or requiring manual confirmation.

## Stale bufferUsername after token change

The `bufferUsername` is fetched once during `startWorkflow` and cached on the `RunningWorkflow` object. The guard is `if (!rw.bufferUsername && rw.config.bufferAccessToken)` — once set, it's never cleared. When `updateWorkflow` changes the Buffer access token, it rebuilds the publisher but doesn't reset `bufferUsername`. The workflow may restart (if it was running), but `!rw.bufferUsername` is already false, so the stale value persists in all subsequent activity log entries. Fix: clear `rw.bufferUsername` in `updateWorkflow` when `bufferAccessToken` changes.

## Test coverage gaps

The `automation-service.test.ts` mock adds `fetchAllRowNumbers` to keep compilation working. But none of the new behavioral logic in `workflow-manager.ts` (pre-marking rows on create, resetting on sheet-ID change, caching Buffer username) is tested. The `testConnection` static method, the `/api/workflows/:id/credentials` endpoint, and the `/api/buffer-test` endpoint have no tests.
